Habit Tracker App 

1. Brief Description 

Building durable habits requires more than intention — it requires a tool that captures what was committed to, holds the user to it on the right days, accommodates real life (illness, travel), nudges them when they forget, and recognizes their progress along the way. 

The task is to build a habit-tracking application where each user can define habits with targets and schedules, mark daily completions, configure reminders, and earn badges for hitting milestones. The system must handle scheduled exceptions gracefully so that legitimate off-days do not break otherwise-strong streaks. 

The goal is a working end-to-end product. Streak logic that correctly respects each habit’s schedule and exceptions is the most important correctness boundary; visual polish and analytics depth are secondary. 

Context & constraints 

Time-box: 6 hours of focused build time. 

AI assistance is expected. Any AI coding assistant (Copilot, Cursor, Claude Code, ChatGPT, etc.) may be used freely. Scope and quality expectations are calibrated to AI-assisted development. 

Tech stack: fully open. Any language, framework, or storage layer is acceptable. 

UI polish is welcomed but secondary — clarity, correctness (especially of streak math), and usability are primary. 

Deliverable: a public GitHub repository with a README (see Submission section). 

2. Business Requirements 

BR1. Provide each user with a private, personal space for tracking habits — only that user’s habits, check-ins, reminders, and badges are visible to them. 

BR2. Allow habits to be defined with a specific target (yes/no completion or a measurable count) so that progress can be evaluated objectively. 

BR3. Allow habits to follow a schedule (daily, specific days of the week, or a flexible weekly count) so each habit only counts on its intended days. 

BR4. Allow scheduled exceptions (e.g. vacation, illness) so legitimate off-days do not break streaks or count as misses. 

BR5. Provide a low-friction way to record daily completions and to look back at the history. 

BR6. Show meaningful progress metrics — current streak, longest streak, completion rate, total completions — for each habit. 

BR7. Allow each habit to have a reminder so the user is prompted on relevant days. External delivery (push, email, SMS) is not required; surfacing in the application is sufficient. 

BR8. Recognize user progress with a set of badges earned automatically when milestones are reached. 

BR9. Authenticate users and ensure no user can see or modify another user’s data. 

BR10. Persist all data reliably so habits, check-ins, exceptions, reminders, and badges are not lost between sessions or restarts. 

3. Functional Requirements 

3.1 Authentication 

FR1. Users can register a new account by providing an email and password. 

FR2. Users can sign in with their email and password. 

FR3. Users can sign out, ending their session. 

FR4. All screens and data are accessible only to authenticated users; unauthenticated requests are redirected to sign-in or rejected. 

FR5. Passwords must be stored hashed (e.g. bcrypt, argon2, scrypt). Plaintext or reversibly-encrypted password storage is not acceptable. 

FR6. On every read and write, the application must ensure a user can only see or modify their own habits, check-ins, exceptions, reminders, and badges. No endpoint may leak another user’s data. 

3.2 Habit management 

FR7. A user can create a habit by providing: name, optional description, optional category, target type, target value (when applicable), schedule, start date, and optional end date. 

FR8. Target type is one of: “Yes/No” (binary completion) or “Count” (a numeric amount, e.g. 30 minutes of reading). For Count-type habits, the user supplies a target value (positive number) and a unit label (free text, e.g. “minutes”, “pages”). 

FR9. Schedule is one of: (a) Daily — every day; (b) Specific weekdays — a chosen subset of Monday–Sunday; (c) Weekly count — a target number of completions per week (1–7), any days. 

FR10. A user can view a list of their habits with at-a-glance progress information (current streak and whether today’s expected completion is done). 

FR11. A user can open a habit to view its full configuration, recent history, and badge eligibility. 

FR12. A user can edit any habit field, including target and schedule. 

FR13. A user can pause, resume, or archive a habit. An archived habit is hidden from default views but its check-in history is preserved and accessible. 

FR14. A user can delete a habit. Deletion cascades to its check-ins, exceptions, and reminder. 

3.3 Exceptions 

FR15. A user can add an exception for a specific date on a habit (e.g. travelling, illness). An exception captures: date, optional reason. 

FR16. An exception day is treated as “not expected” for that habit: it does not contribute to scheduled days for streak or completion-rate purposes, and missing a check-in on an exception day does not break the streak. 

FR17. Exceptions can be edited (reason) or removed. 

3.4 Check-ins 

FR18. A user can record a check-in for a habit on a specific date. The default date is today; past dates within the last 7 days are allowed (out-of-range backdating is rejected). Each check-in captures: date, status (Done / Partial / Skipped), value (for Count-type habits), optional note. 

FR19. A user has at most one check-in per habit per date. Submitting a check-in on a date that already has one must update the existing record rather than create a duplicate. 

FR20. A user can edit or remove their own check-in. Removing a check-in must recompute the affected streak and completion-rate values. 

3.5 Streaks and progress 

FR21. For each habit, the application must compute and display: current streak, longest streak ever, total completions (Done status), and completion rate over the last 30 days (Done count divided by scheduled, non-exception days in the window). 

FR22. Streak logic must respect the habit’s schedule (only scheduled days count) and its exceptions (treated as not-expected). For Daily and Specific-weekdays schedules, the current streak is the count of consecutive scheduled, non-exception days ending on or immediately before today with a Done check-in. For a Weekly-count schedule, the streak is measured in consecutive weeks meeting the target. 

FR23. Each habit detail view must include a simple history view of recent days (e.g. the last 30 days as a heatmap, calendar grid, or list). 

3.6 Reminders / alerts 

FR24. A user can configure at most one reminder per habit with: time of day (HH:MM in the user’s local time), enabled toggle. The reminder may be added, edited, or removed at any time. 

FR25. The application must surface today’s due habits (habits scheduled today, not on exception, without a Done check-in) in a dedicated panel on the main dashboard. External delivery of alerts (push, email, SMS) is NOT required for this assessment. 

3.7 Badges 

FR26. The application defines a small fixed set of badges with clear earning criteria (e.g. First check-in; 7-day streak; 30-day streak; 100 total completions; 5 active habits). The exact set is the candidate’s choice but must be stated in the README and visible to the user. 

FR27. Badges are awarded automatically the moment the corresponding criterion is met. Once earned, a badge stays earned and cannot be revoked, even if the criterion is no longer met. 

FR28. A badges view lists the user’s earned badges (with earned-at timestamp) and the criteria for unearned ones. 

3.8 Listing, filtering, sorting, pagination 

FR29. The habits list can be filtered by status (Active / Paused / Archived) and sorted by name and by most-recent check-in. 

FR30. The check-in history (per habit and, optionally, a global recent-activity feed) must be paginated with a sensible default page size (e.g. 30). 

FR31. Other lists (habits, badges, exceptions per habit) are bounded for a single user and need not paginate, but must remain responsive at expected sizes (e.g. 50+ habits). 

3.9 Persistence 

FR32. All data — users, habits, exceptions, check-ins, reminders, badges, and badge awards — must persist across application restarts. Any storage approach is acceptable provided the choice is justified in the README. 

3.10 Validation 

FR33. Habit name is required and must not be empty or whitespace-only. 

FR34. For Count-type habits, target value must be a positive number and a unit label must be supplied. 

FR35. Schedule must be valid: Specific-weekdays requires at least one selected day; Weekly-count target must be between 1 and 7. 

FR36. Reminder time, when set, must be a valid HH:MM (24-hour) string. 

FR37. All validation failures must be communicated to the user with a clear, actionable message. 

4. Acceptance Criteria 

AC1. A new user can register, sign out, and sign back in successfully. Passwords are stored hashed (verifiable by inspecting the storage or code). 

AC2. An unauthenticated visitor cannot reach any application screen or API endpoint without being redirected to sign-in or rejected. 

AC3. A signed-in user can create a Yes/No habit with a chosen schedule. The habit appears in their habit list immediately. 

AC4. A signed-in user can create a Count habit with a target value and unit. The habit list reflects today’s target and today’s recorded value. 

AC5. A user cannot see or modify another user’s habits, check-ins, exceptions, reminders, or badges through any endpoint. 

AC6. A user can record a Done check-in for today. The habit’s current streak increases accordingly and persists after restart. 

AC7. Re-submitting a check-in for a date that already has one updates the existing record rather than creating a duplicate (one check-in per habit per date). 

AC8. A user can edit or remove a check-in; the affected streak and completion-rate values recompute correctly. 

AC9. Daily habit completed for 7 consecutive days shows a current streak of 7. Missing a non-exception scheduled day breaks the streak. 

AC10. Specific-weekdays habit (e.g. Mon/Wed/Fri): the streak only counts scheduled days. Not checking in on a Tuesday does not affect the streak. 

AC11. Weekly-count habit (e.g. 3 per week): a week with 3 or more Done check-ins counts as a streak week. Consecutive successful weeks accumulate the streak. 

AC12. Adding an exception for a date prevents that day from being counted as a missed scheduled day; the streak is preserved across exception days. 

AC13. The dashboard surfaces today’s due habits (scheduled today, not on exception, without a Done check-in) in a dedicated panel. 

AC14. A reminder time can be set on a habit and is shown on the habit detail and on today’s panel. (External notifications are out of scope.) 

AC15. Completing the first ever check-in immediately awards the “First check-in” badge. Achieving a 7-day streak immediately awards the corresponding streak badge. 

AC16. Earned badges are visible on the badges view alongside their earned-at timestamp; unearned badges display their criteria. 

AC17. The habit list can be filtered by status (Active / Paused / Archived) and sorted as specified; archived habits are hidden from the default view but their history is intact. 

AC18. The check-in history view paginates correctly across pages with the chosen page size. 

AC19. All data (users, habits, exceptions, check-ins, reminders, badges) persists after stopping and restarting the application. 

5. Submission Requirements 

The candidate must submit a link to a public GitHub repository. The repository must include a README that contains the following sections. 

README must include 

Setup & run instructions. Any reviewer should be able to clone the repo and run the app locally in under 10 minutes. 

Tech stack & rationale. What was chosen and why, particularly the storage layer and the auth/password-hashing approach. 

Architectural overview. A short description of how the code is organized (layers, modules, key files). 

How AI tools were used. Which AI assistants were used; which parts of the codebase were AI-generated, AI-assisted, or hand-written; what was reviewed, edited, or rejected; anything surprising or worth flagging. 

Assumptions. Anything assumed about scope, users, or environment. 

Trade-offs. What was deliberately deprioritized and why, given the 6-hour budget. 

Future work. What would be added or improved with more time. 

Required deliverables (beyond the README) 

Incremental commit history. Commits should be incremental and meaningful so that reviewers can see how the work progressed. Single-commit submissions are discouraged. 

What is not required 

Production deployment. Local-run is sufficient. 

External notifications (push, email, SMS). In-app surfacing of due habits is sufficient. 

Mobile-native apps. 

Multi-device real-time sync. 

Social features (friends, sharing, leaderboards). 

Habit templates, suggested habits, or content libraries. 

Custom badge creation by the user. A small predefined badge set is fine. 

Calendar export or iCal integration. 

Streak-freeze / second-chance mechanics. 

Multi-tenant or organization-level separation. 

What is welcomed but optional 

A calendar or heatmap visualization of each habit’s history. 

Tagging or categorization of habits beyond a simple category field. 

A global recent-activity feed across all habits. 

Personal data export (CSV or JSON). 

Aggregate stats (e.g. overall completion rate this week, top habits, etc.). 

Timezone handling beyond the user’s local time. 

Containerization (Dockerfile, docker-compose). 

Keyboard shortcuts, dark mode, or other UX niceties.
